Optimize Northern Shield for solo developer sustainability.

 CONTEXT:
 - Solo project, fast iteration, no team, no CI
 - src/ is the live codebase — this is what runs
 - engine/ contains old skeleton files — dead code, ignore it
 - prompts/ contains AI guidance prompts — update them when the project changes significantly
 - Norse dark fantasy visual theme is established — maintain it consistently

 THE CODEBASE SHOULD:
 - Be understandable after a 2-week break without re-reading all files
 - Minimize mental overhead — one clear place for each type of logic
 - Avoid magical behavior (no hidden side effects, no framework magic)
 - Support fast debugging (console.log + browser DevTools is enough)
 - Support safe experimentation (changing ENEMY_DEFS values should be risk-free)

 PRACTICAL RULES:
 - game.js is the single integration point — keep all game state here
 - All tunable numbers live in ENEMY_DEFS / TOWER_DEFS — never hardcode stats inline
 - If a function grows beyond ~40 lines, consider splitting it (but only if it improves clarity)
 - Name things from the player's perspective: gold not credits, slain not kills
 - UI names vs internal keys: Graveborn/Wisp/Golem/Banshee in labels,
     infantry/drone/tank/banshee in code keys
 - Visual theme: Norse dark fantasy — deep purples, warm golds, dramatic glows
     Don't drift from this without intent

 SIGNS OF DRIFT TO WATCH FOR:
 - A function that does two unrelated things
 - A variable that means different things in different contexts
 - A "temporary" workaround that's still there 3 sessions later
 - Copy-pasted draw logic that should be shared (e.g. drawSpriteFrame was added for this)
